# Week 8: 长上下文管理与安全工程

## 🎯 目标
攻克长上下文瓶颈，构建防注入、脱敏、审计管道。掌握 KV Cache 内存 profiling、上下文压缩策略、OWASP LLM Top 10 防御、PII 自动化脱敏、工具沙箱执行、红队对抗测试及数据生命周期合规管理，交付高可用、可审计、抗攻击的生产级 AI 服务。

## Day 1：Token 爆炸 / 注意力稀释 / KV Cache 瓶颈分析
- **目标**：量化长上下文下的显存膨胀、计算延迟与注意力衰减规律，定位性能拐点。
- **实操**：
  1. 使用 `vLLM` 或 `transformers` 内置 Profiler，记录 1K~128K 上下文下的 KV Cache 显存占用、Prefill/Decode 延迟、GPU 利用率。
  2. 运行 Needle-in-a-Haystack 基准测试，绘制注意力热力图，量化长距离信息召回率衰减曲线。
  3. 分析 `PagedAttention` 分页策略对碎片化内存的影响，对比 `block_size=16/32/64` 的 Cache 分配效率。
  4. 输出瓶颈诊断报告：显存线性/指数增长拐点、注意力稀释阈值、Prefill 阻塞根因。
- **Prompt 模板**：
```md
编写长上下文性能 Profiling 与注意力分析脚本，要求：
1. 自动扫描 1K~128K 上下文长度，记录 VRAM 峰值、TTFT、TPOT、GPU Util
2. 运行 Needle-in-a-Haystack 测试，提取关键实体召回率，绘制注意力热力图
3. 对比 block_size [16,32,64] 下的 PagedAttention 碎片率与 OOM 边界
4. 输出 Markdown 诊断报告：包含显存增长曲线、注意力稀释拐点、Prefill 阻塞归因
5. 内置安全熔断：单条测试超时 >30s 或 OOM 自动跳过，记录异常栈
6. 支持导出 CSV/JSON 原始数据，附 Matplotlib/Plotly 可视化脚本
```
- **验收**：
  - ✅ 显存/延迟增长曲线完整，定位性能拐点（如 32K/64K）
  - ✅ 注意力热力图清晰，长距离召回率衰减可量化
  - ✅ PagedAttention 碎片率分析完整，block_size 推荐明确
  - ✅ 诊断报告结构化，支持复现与横向对比

## Day 2：上下文压缩 (滑动窗口 + 摘要链)
- **目标**：实现高保真上下文管理，平衡记忆保留与计算开销。
- **实操**：
  1. 构建 `ContextManager`：滑动窗口保留最近 N 轮对话，超窗旧消息进入摘要队列。
  2. 实现递归摘要链：LLM 按主题聚类旧对话 → 生成结构化摘要 → 追加至 System Prompt。
  3. 注入元数据标签：保留关键实体、待办事项、用户偏好，避免摘要丢失决策依据。
  4. 评估压缩效果：使用 BERTScore + ROUGE-L + LLM 意图对齐评分，验证信息保真度。
- **Prompt 模板**：
```md
实现高保真上下文压缩流水线，要求：
1. 滑动窗口机制：可配置 window_size，超窗消息自动入队待压缩
2. 摘要链生成：LLM 按主题聚类，输出结构化摘要（事实/意图/状态/约束）
3. 元数据保留：关键实体/偏好/未完成事项独立提取，防止摘要过滤丢失
4. 质量评估：自动计算 BERTScore/ROUGE-L，LLM 意图对齐评分 >0.9 为合格
5. 无缝集成：FastAPI 路由自动调用 ContextManager，返回压缩后 messages 列表
6. 支持热切换：摘要策略/窗口大小运行时可调，无需重启服务
```
- **验收**：
  - ✅ 上下文压缩比 ≥3x，显存/延迟显著下降
  - ✅ 意图保留率 >90%（LLM 对齐评分验证）
  - ✅ 摘要结构化完整，关键实体零丢失
  - ✅ 无缝集成至对话链路，零断言/格式错误

## Day 3：OWASP LLM Top 10 防御 (Prompt 注入/越权)
- **目标**：构建输入/输出双向防护网，阻断提示词注入与权限越权攻击。
- **实操**：
  1. 对照 OWASP LLM Top 10，聚焦 LLM01 (Prompt Injection) 与 LLM02 (Insecure Output Handling)。
  2. 实现输入清洗层：分隔符强校验、系统指令隔离、角色/权限白名单匹配、恶意模式正则过滤。
  3. 部署输出防护：JSON Schema 强校验、危险指令拦截（如执行 shell/读写敏感路径/伪造用户身份）。
  4. 集成轻量分类器：本地小型 LLM 或规则引擎实时评分，>0.8 威胁值直接拦截并记录。
- **Prompt 模板**：
```md
构建 OWASP LLM Top 10 防御中间件，要求：
1. 输入防护：系统指令强隔离（<|system|> 分隔），角色/权限白名单校验
2. 恶意模式拦截：正则+词向量混合过滤，覆盖 DAN、越权指令、隐式 Prompt 注入
3. 输出护栏：严格 JSON Schema 校验，危险操作（代码执行/文件读写/身份伪造）自动拦截
4. 轻量分类器：本地规则引擎/小模型实时打分，威胁阈值可配，误杀率 <2%
5. 拦截响应：返回标准化 403 Payload，附拒绝原因与 Trace ID
6. 全量日志：记录拦截类型、原始 Prompt 哈希、分类器置信度，支持审计导出
```
- **验收**：
  - ✅ 注入拦截率 >95%（覆盖 OWASP 测试集）
  - ✅ 误杀率 <2%，正常业务 Prompt 零阻断
  - ✅ 输出 Schema 校验 100% 生效，越权指令 100% 拦截
  - ✅ 防御中间件延迟 <10ms，支持热更新规则

## Day 4：PII 脱敏 (正则/NLP) + 审计日志
- **目标**：实现自动化隐私保护与全链路可追溯审计。
- **实操**：
  1. 集成 `Microsoft Presidio` 或 `spaCy NER`，识别身份证、手机号、邮箱、地址、银行卡等 PII。
  2. 配置脱敏策略：哈希替换、部分掩码（如 `138****5678`）、Token 化映射（可逆用于内部排查）。
  3. 构建结构化审计日志：JSON 格式记录 `request_id`, `timestamp`, `user_id`, `action`, `pii_detected`, `sanitized_payload`, `llm_response_hash`。
  4. 日志持久化：写入只追加存储（如 MinIO/ES/PostgreSQL Audit Table），支持按 Trace ID 全链路检索。
- **Prompt 模板**：
```md
实现 PII 自动脱敏与结构化审计流水线，要求：
1. 集成 Presidio/spaCy NER，支持 10+ 常见 PII 类型识别，召回率 >98%
2. 脱敏策略可配：掩码/哈希/Token 映射，支持白名单实体跳过脱敏
3. 审计日志：JSON 格式，包含 trace_id, timestamp, user, action, pii_count, payload_hash
4. 存储策略：只追加写入（ES/MinIO/DB），支持按 trace_id 秒级检索，保留 90 天
5. 性能要求：脱敏+日志写入总开销 <5% 请求延迟，异步批量刷新
6. 合规导出：一键生成 GDPR/PIPL 合规审计报告（脱敏率、PII 分布、访问记录）
```
- **验收**：
  - ✅ PII 识别召回率 >98%，误标率 <1%
  - ✅ 脱敏策略生效，敏感数据零明文落盘
  - ✅ 审计日志结构化完整，Trace ID 全链路可追溯
  - ✅ 日志写入延迟 <5%，支持秒级检索与合规导出

## Day 5：安全中间件 (请求过滤/沙箱执行)
- **目标**：实现运行时安全隔离，防止恶意工具调用与代码注入。
- **实操**：
  1. 封装 FastAPI 安全中间件：前置校验 Payload 体积、MIME 类型、参数类型、注入特征。
  2. 构建工具执行沙箱：基于 `Docker` 或 `gVisor` 隔离网络/文件系统，限制 CPU/内存/超时。
  3. 实现安全执行协议：工具调用前校验权限，执行中监控 syscall，异常自动 kill 并记录堆栈。
  4. 响应安全组装：过滤 LLM 返回的未授权外部链接、可执行脚本、隐藏 Markdown 注入。
- **Prompt 模板**：
```md
编写安全中间件与工具沙箱执行引擎，要求：
1. 请求过滤：拦截超大小 Payload、非法 MIME、类型不匹配、已知注入特征
2. 沙箱隔离：Docker/gVisor 容器隔离，禁用网络/根目录写入，限制 CPU/内存/执行时间
3. 工具调用安全：权限校验前置，执行中 syscall 监控，超时/越权自动 kill
4. 响应净化：过滤未授权 URL、可执行代码块、隐藏 XSS/Markdown 注入
5. 指标暴露：sandbox_executions_total, blocks_total, latency, violations, 对接 Prometheus
6. 优雅降级：沙箱不可用时自动 fallback 至规则校验模式，不中断服务
```
- **验收**：
  - ✅ 恶意 Payload 100% 前置拦截，沙箱越权执行 0 成功
  - ✅ 工具调用权限校验完整，响应净化无 XSS/注入残留
  - ✅ 沙箱开销可控（冷启动 <1s，执行延迟 <200ms）
  - ✅ 全链路安全指标暴露，支持实时监控与告警

## Day 7：安全白皮书 + 数据保留/擦除策略
- **目标**：输出合规文档与数据生命周期管理机制，满足隐私法规要求。
- **实操**：
  1. 撰写安全架构白皮书：覆盖威胁模型、防御架构、数据流向、合规映射（PIPL/GDPR/ISO27001）。
  2. 实现数据保留策略：按数据类型设置 TTL（对话日志 90 天、审计日志 180 天、PII 脱敏副本 30 天）。
  3. 构建自动化擦除管道：支持用户级/会话级硬删除、加密擦除（Crypto-Shredding）、级联清理关联数据。
  4. 合规验证：模拟数据主体删除请求（DSAR），验证全链路擦除完整性与不可恢复性。
- **Prompt 模板**：
```md
生成安全白皮书与数据生命周期管理管道，要求：
1. 白皮书结构：威胁模型、防御架构、数据流向图、合规映射表（PIPL/GDPR）、红队结论
2. 保留策略：按数据类型配置 TTL，到期自动归档或清理，支持合规豁免审批流
3. 擦除管道：用户级/会话级硬删除，加密擦除（密钥销毁），级联清理 KV Cache/审计/备份
4. 合规验证：模拟 DSAR 请求，验证全链路数据不可恢复，输出擦除证明报告
5. 自动化调度：Cron/队列触发清理任务，失败重试，执行日志可审计
6. 一键导出：生成符合审计要求的合规包（白皮书 + 策略配置 + 擦除证明 + 日志快照）
```
- **验收**：
  - ✅ 安全白皮书完整，覆盖架构/威胁/合规映射
  - ✅ TTL 策略自动执行，过期数据 100% 清理或归档
  - ✅ 擦除管道生效，DSAR 模拟验证全链路不可恢复
  - ✅ 合规审计包一键生成，满足监管审查要求

## 动态调整建议
- **无 GPU 资源做 Profiling**：
  - 改用 HuggingFace `transformers` + CPU 模式运行 1K~8K 长度测试，聚焦注意力衰减趋势而非绝对延迟
  - 使用公开数据集（如 LongBench）替代自定义 Needle 测试
- **安全合规经验不足**：
  - 优先集成现成库（`Microsoft Presidio`, `NeMo Guardrails`, `OWASP LLM Top 10 Checklist`），快速跑通 D3/D4
  - 白皮书可基于开源模板（如 NIST AI RMF）填充，聚焦数据流向与脱敏策略
- **沙箱部署复杂**：
  - D5 可先用 `subprocess` + `seccomp` + `cgroups` 实现轻量级隔离，验证逻辑后再迁移至 Docker/gVisor
  - 网络/磁盘限制通过 `iptables` + `readonly mount` 替代，降低运维门槛
- **红队用例难以构造**：
  - 使用开源红队工具（`Promptfoo`, `Garak`, `LLM Guard`）生成初始攻击集，人工补充业务特定场景
  - 拦截率不达标时，优先优化“系统提示隔离”与“输出 JSON Schema 强校验”，收益最高
- **团队偏向 Java/Go**：
  - 防御/脱敏/沙箱架构完全可迁移：Java 用 `Spring AI` + `OpenPolicyAgent` + `gVisor Java SDK`；Go 用 `Gin 中间件` + `Presidio Go` + `containerd`
