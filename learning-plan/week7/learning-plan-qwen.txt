## 🎯 目标
优化高并发流式架构，实现连接池、降级策略与压测验证。掌握异步 I/O 深度调优、SSE/WebSocket 生产化、任务异步解耦、精准限流熔断、500 VU 压测闭环及 Nginx/Gunicorn 生产级部署，构建高可用、低延迟、可自愈的 AI API 服务。

## Day 1：FastAPI 异步深度优化 (asyncpg/redis 连接池)
- **目标**：掌握异步数据库与缓存连接池管理，消除事件循环阻塞，提升 I/O 吞吐。
- **实操**：
  1. 集成 `asyncpg` (PostgreSQL) 与 `redis.asyncio`，配置连接池 `min_size/max_size/max_inactive_connection_lifetime`。
  2. 使用 `Depends` 实现会话级生命周期管理，确保请求结束后自动 `release()`，杜绝连接泄漏。
  3. 全局扫描替换同步阻塞调用（如 `requests`, `time.sleep`, 同步 ORM），强制使用 `aiohttp`/`httpx`/异步驱动。
  4. 注入异步中间件：SQL 查询超时控制（默认 3s）、慢查询日志、连接池水位监控 (`pool.size`, `pool.free_size`)。
- **Prompt 模板**：
```md
编写生产级 FastAPI 异步 I/O 优化方案，要求：
1. 集成 asyncpg + redis.asyncio 连接池，支持动态 min/max 配置与心跳探测
2. 使用 Depends 实现连接会话自动 acquire/release，杜绝连接泄漏与事件循环阻塞
3. 全链路异步化改造：替换所有同步阻塞调用，提供异步兼容包装层
4. 注入查询超时中间件，超时自动 cancel 任务并记录慢查询 TraceID
5. 暴露连接池指标：active/idle/max/queue_depth，支持 Prometheus 自动抓取
6. 提供压测对比报告：优化前后 QPS、P95 延迟、事件循环阻塞告警次数
```
- **验收**：
  - ✅ 连接池健康稳定，零泄漏/零阻塞事件循环
  - ✅ DB/Redis 查询 P95 < 50ms，超时自动熔断
  - ✅ 暴露完整连接池指标，Grafana 实时可视
  - ✅ 异步化覆盖率 100%，同步调用拦截告警生效

## Day 2：SSE/WebSocket 生产化 (心跳/重连/保序)
- **目标**：实现工业级流式传输协议，保障断线重连、消息保序与零丢失。
- **实操**：
  1. 规范 SSE 响应头：`Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`。
  2. 实现心跳保活：每 15s 发送 `: ping` 注释包，维持代理层/客户端连接活跃，防 Nginx/CDN 超时断开。
  3. 客户端重连机制：指数退避 (1s→2s→4s→8s) + `Last-Event-ID` 断点续传，服务端校验 ID 防重复投递。
  4. 消息保序与去重：基于 `event_id` 单调递增 + 客户端滑动窗口去重，处理弱网乱序/重复 chunk。
  5. 背压控制：服务端生成队列限流（如 1000 chunks），满额时暂停推理，避免内存爆炸。
- **Prompt 模板**：
```md
构建生产级 SSE/WebSocket 流式传输架构，要求：
1. 严格遵循 SSE 规范：event/data/retry/id 字段完整，心跳 :ping 每 15s 发送
2. 客户端重连：指数退避策略，携带 Last-Event-ID 实现断点续传
3. 保序与去重：服务端单调 event_id 递增，客户端滑动窗口过滤重复/乱序 chunk
4. 背压控制：生成队列限流 1000 chunk，满额暂停推理，队列释放后恢复
5. 网络抖动模拟：断线 3s/10s 重连测试，验证消息完整性、顺序与零丢失
6. 日志追踪：每条流绑定 trace_id，记录 chunk 发送耗时与客户端 ACK 状态
```
- **验收**：
  - ✅ 心跳稳定不断连，代理层不主动关闭流
  - ✅ 断线重连 <3s，基于 Last-Event-ID 零消息丢失/乱序
  - ✅ 背压机制生效，服务端内存水位平稳
  - ✅ 全量流式 trace 可审计，chunk 投递延迟 <100ms

## Day 3：Celery/RQ 解耦长耗时任务
- **目标**：将阻塞性长任务移出请求链路，实现异步执行、状态追踪与弹性扩容。
- **实操**：
  1. 部署 Celery (或 RQ) + Redis/RabbitMQ Broker，配置 Worker 并发数与 Prefetch 策略。
  2. 抽象长任务接口：文件解析、向量入库、大模型批量推理、异步通知推送。
  3. 实现任务状态机：`PENDING -> STARTED -> SUCCESS/FAILURE/RETRY`，FastAPI 提供 `/tasks/{id}` 状态查询。
  4. 配置重试与幂等：指数退避重试 (`max_retries=3`)，基于 `task_id` + 分布式锁防重复执行。
  5. 监控队列深度与 Worker 健康，配置告警阈值（积压 >500 或 连续失败率 >10%）。
- **Prompt 模板**：
```md
编写 Celery/RQ 异步任务解耦流水线，要求：
1. 配置 Broker/Backend，支持 Redis/RabbitMQ 热切换，Worker 自动发现任务
2. 定义标准任务接口：支持 @task(bind=True) 绑定上下文，返回 TaskResult 状态
3. FastAPI 暴露任务提交与查询端点，支持 SSE 推送任务进度更新
4. 重试与幂等：指数退避 + 唯一 task_id，重复提交自动复用已存在结果
5. 监控告警：队列深度、Worker 存活、失败率、执行耗时，对接 Prometheus
6. 优雅停机：SIGTERM 捕获，等待 in-flight 任务完成再退出，支持 graceful shutdown
```
- **验收**：
  - ✅ 请求链路响应时间 < 200ms，长任务 100% 异步化
  - ✅ 任务状态可追踪，SSE 进度推送实时准确
  - ✅ 重试/幂等验证通过，无重复执行/数据污染
  - ✅ 队列监控完整，积压/失败自动告警

## Day 4：Redis 滑动窗口限流 + IP 封禁
- **目标**：实现精准、低延迟的流量治理与恶意请求拦截。
- **实操**：
  1. 使用 Redis `ZSET` + Lua 脚本实现滑动窗口限流，支持多维策略（全局/IP/用户/路由）。
  2. 配置动态阈值：常规请求 100 req/min，AI 生成 20 req/min，VIP 通道独立配额。
  3. 实现 IP 封禁逻辑：连续 5xx >5 次 或 暴力扫描特征触发自动 ban，支持 TTL 与申诉白名单。
  4. 封装 FastAPI 中间件：请求前置拦截，命中限流返回 `429 Too Many Requests` + `Retry-After` 头。
  5. 限流指标上报：`rate_limit_hits_total`, `ip_ban_count`, `quota_remaining`，支持 Grafana 面板。
- **Prompt 模板**：
```md
实现 Redis 滑动窗口限流与动态 IP 封禁中间件，要求：
1. ZSET + Lua 原子操作实现滑动窗口，精度 <100ms，支持多维 Key 隔离
2. 分级配额：global/ip/user/route 四级策略，配置热加载无重启
3. IP 封禁：连续 5xx≥5 或异常 UA 模式触发 ban，TTL 可配，支持内部白名单绕过
4. 中间件拦截：前置校验，429 响应带 Retry-After 与 X-RateLimit-* 头
5. 指标暴露：限流命中数、封禁 IP 列表、配额余量，对接 Prometheus
6. 性能要求：单请求限流开销 <2ms，不阻塞异步事件循环
```
- **验收**：
  - ✅ 滑动窗口限流精准，误差 <1s
  - ✅ 动态封禁自动触发，白名单/内部流量 100% 绕过
  - ✅ 429 响应符合 RFC，携带标准 Rate-Limit 头
  - ✅ 限流中间件开销 <2ms，指标实时上报

## Day 6：降级与熔断 (OOM 自动降级至缓存/轻量模型)
- **目标**：构建系统级熔断与降级机制，保障极端负载下的服务可用性。
- **实操**：
  1. 集成熔断器逻辑（如 `pybreaker` 或自定义状态机）：`Closed -> Open (触发阈值) -> Half-Open (试探恢复)`。
  2. 资源监控探针：CPU >90%、RAM >85%、GPU VRAM >92%、队列积压 >1000 触发熔断。
  3. 降级策略：OOM 时自动切换至 Redis 缓存响应（TTL 可配）或轻量模型（1.5B/3B 蒸馏版），保持接口兼容。
  4. 客户端无感降级：返回 `X-Fallback-Mode: cache/lightweight` 头，前端提示“当前为简化模式”。
  5. 熔断日志与指标：记录状态切换、降级命中率、恢复耗时，支持告警与自动回切。
- **Prompt 模板**：
```md
构建全链路熔断与智能降级系统，要求：
1. 状态机实现 Closed/Open/Half-Open，失败率/延迟双阈值触发，自动回切
2. 资源探针：CPU/RAM/GPU/队列深度实时监控，超阈值立即熔断
3. 降级策略：自动 fallback 至 Redis 缓存 或 轻量模型，接口签名/格式完全兼容
4. 客户端透传：X-Fallback-Mode 头标识降级状态，前端优雅提示
5. 指标与告警：熔断触发次数、降级命中率、恢复耗时，对接 Prometheus/钉钉/企微
6. 安全机制：降级期间屏蔽非核心写操作，保护核心数据一致性
```
- **验收**：
  - ✅ 熔断触发准确，降级切换 <500ms
  - ✅ 降级响应可用，客户端无 5xx 崩溃
  - ✅ Half-Open 状态自动恢复，无震荡切换
  - ✅ 熔断/降级全链路指标可审计，告警及时

## Day 7：Uvicorn/Gunicorn/Nginx 生产配置
- **目标**：完成生产级部署栈硬化，优化进程管理、反向代理、安全与可观测性。
- **实操**：
  1. Gunicorn 配置：`uvicorn.workers.UvicornWorker`，`workers=2*CPU+1`，`keepalive=65`，`backlog=2048`。
  2. Nginx 调优：关闭 `proxy_buffering` 保障 SSE/WS 实时性，配置 `chunked_transfer_encoding`，TLS 1.3 + HSTS。
  3. 安全策略：CORS 白名单、CSRF 防护、Rate Limit 前置、敏感头隐藏 (`Server`, `X-Powered-By`)。
  4. 健康检查：`/healthz` 端点返回 `200 OK`，Nginx `location /health` 用于负载均衡探测与自动剔除故障节点。
  5. 容器化交付：多阶段 Dockerfile，非 root 运行，`docker-compose` 编排 `app + redis + postgres + nginx`。
- **Prompt 模板**：
```md
编写 Uvicorn/Gunicorn/Nginx 生产级部署配置，要求：
1. Gunicorn 调优：UvicornWorker, workers 按核数公式, backlog/keepalive/graceful_timeout 配置
2. Nginx 反向代理：proxy_buffering off, chunked_transfer_encoding on, 代理头透传, TLS 1.3/HSTS
3. 安全加固：CORS 白名单, 隐藏 Server/X-Powered-By, 基础 Rate Limit, CSP 头
4. 健康检查：/healthz 端点，返回 DB/Redis/Worker 状态，Nginx upstream 自动剔除故障节点
5. 容器化：多阶段 Dockerfile, 非 root 运行, docker-compose 编排全栈, 日志 JSON 格式化
6. 启动脚本：entrypoint.sh 自动检测依赖、预热连接池、优雅启停
```
- **验收**：
  - ✅ Gunicorn/Nginx 启动零报错，Workers 按预期分配
  - ✅ SSE/WS 通过代理零延迟、零断流
  - ✅ 安全头/健康检查/日志格式化符合生产规范
  - ✅ `docker-compose up` 一键拉起全栈，压测指标达标

## 动态调整建议
- **资源受限（单核 <4GB RAM）**：
  - D3 优先使用 `RQ` 替代 Celery，内存开销更低，配置更简单
  - D5 压测降至 200 VU，聚焦延迟与错误率趋势而非绝对峰值
- **无高并发经验**：
  - D1/D2 先跑通基础异步与 SSE，D4/D6 限流熔断可先用 `slowapi`/`tenacity` 现成库快速落地
  - 重点理解“背压”与“熔断状态机”思想，而非死磕底层源码
- **模型推理侧负载极重**：
  - D6 降级优先对接 Week 5 导出的 `GGUF Q4` 轻量版，或 Week 4 的 `vLLM 低并发实例`，保持业务连续性
  - 缓存层可预先计算高频 FAQ 或模板回复，TTL 设置 5-15min
- **前端/客户端联调频繁**：
  - D2 重连与 D6 降级策略需与前端对齐 `Retry-After` 与 `X-Fallback-Mode` 解析逻辑，避免前端误报异常
- **团队偏向 Go/Java**：
  - 核心架构模式（滑动窗口限流、熔断状态机、背压队列）可无缝迁移至 `gin/redis` 或 `Spring Cloud Gateway + Resilience4j`
